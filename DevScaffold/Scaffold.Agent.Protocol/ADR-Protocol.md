# ADR – Scaffold.Agent.Protocol

**Date:** 2026-03-04
**Updated:** 2026-03-22
**Status:** Accepted
**Affected projects:** Scaffold.Agent.Protocol, Scaffold.ServiceHost, Scaffold.CLI

---

## Context

The `Scaffold.Agent.Protocol` project contains the complete protocol definition for communication between the CLI and the ServiceHost in protobuf format. The `.proto` files serve both as schema definitions and as communication documentation. The decisions recorded here explain why each message contains what it does and why it is structured the way it is.

---

## Decisions

---

### 1. `request_id` correlation identifier in every request

**Decision:** Every CLI-initiated request (`InferRequest`, `LoadModelRequest`, `UnloadModelRequest`, `ListModelsRequest`, `CancelInferRequest`) contains a `request_id` field.

**Rationale:**
- The two pipes are asynchronous — a request and its response event are not guaranteed to arrive sequentially
- The CLI uses `request_id` to match each event to the request that produced it
- GUID format is recommended — generated by the CLI at invocation time
- In `CancelInferRequest`, it identifies which running request should be cancelled

**Note:** `ShutdownRequest` has no `request_id` because after shutdown there is no need for correlation — the ServiceHost is stopping.

---

### 2. `CommandEnvelope` and `EventEnvelope` wrapper messages

**Decision:** The command pipe always carries `CommandEnvelope`, and the event pipe always carries `EventEnvelope` — never the concrete message types directly.

**Rationale:**
- Named Pipe is stream-based — the receiving side needs to know what it is reading
- The `oneof` field guarantees that at most one command or event is active at a time
- In C#, the `CommandCase` / `EventCase` enum enables clean switch-based dispatch
- Extensibility: adding a new command or event requires only a new `oneof` branch without breaking existing code

---

### 3. `oneof` in `CommandEnvelope` and `EventEnvelope`

**Decision:** Both wrapper messages use a `oneof` field to hold the active subtype.

**Rationale:**
- At the binary level, only the active field is serialised — compact format
- Type-safe processing — the `CommandCase` enum precisely identifies what the message contains
- The alternative of a separate pipe per message type would result in an unmanageable number of pipes

---

### 4. `inference.proto` — `InferRequest` contents

**Decision:** `InferRequest` contains: `request_id`, `step_id`, `model_alias`, `system_prompt`, `user_input`, `max_tokens`, `output_folder`.

**Field-by-field rationale:**

| Field | Rationale |
|---|---|
| `request_id` | Correlation identifier for matching events on the event pipe |
| `step_id` | Used for logging and status display — the CLI knows which step is running |
| `model_alias` | The ServiceHost resolves this to a GGUF path via the ModelCache |
| `system_prompt` | The agent's role — sourced from the step agent config YAML |
| `user_input` | Output of `InputAssembler` — the input YAML plus all referenced file contents |
| `max_tokens` | `0` means use the ServiceHost default — flexible, not always required |
| `output_folder` | The step output folder determined by the CLI, including the generation number |

**Why the model path is not included directly:**
- The path is an internal implementation detail of the ServiceHost
- The alias abstraction means the CLI has no knowledge of filesystem structure
- The `models.yaml` mapping lives in one place — easier to maintain

**Why `output_folder` flows from CLI to ServiceHost (not the other way):**
- Computing the generation number is the CLI's responsibility — the CLI knows the project output structure
- The CLI counts existing `{stepId}_*` folders and creates the next one
- The ServiceHost does not need to know about generations — it only uses the folder it receives
- If `output_folder` is empty, the ServiceHost falls back to its own `--output` startup parameter (backward compatibility)
- The CLI receives the actual output file path back via `InferenceCompletedEvent`

---

### 5. `inference.proto` — `CancelInferRequest` contents

**Decision:** `CancelInferRequest` contains only `request_id`.

**Rationale:**
- Minimal message — only identification of which request to cancel is needed
- No other data is required to execute the cancellation

---

### 6. `inference.proto` — `ShutdownRequest` contents

**Decision:** `ShutdownRequest` contains only a `force` bool field.

**Rationale:**
- `force = false`: graceful shutdown — waits for the active inference to complete
- `force = true`: immediate shutdown — cancels the active inference
- No `request_id` — correlation is unnecessary after shutdown

---

### 7. `models.proto` — three commands only

**Decision:** `models.proto` contains three commands: `LoadModelRequest`, `UnloadModelRequest`, `ListModelsRequest`.

**Rationale:**
- `LoadModelRequest`: optional pre-loading — not required when lazy loading, but useful when the model is known to be needed soon
- `UnloadModelRequest`: free memory — when a model will not be needed for a while
- `ListModelsRequest`: lists loaded models only — the human knows which models are available; only the loaded state is relevant at runtime

**Why no additional model commands:**
- Managing `models.yaml` (adding, removing entries) is a human task — not a protocol concern
- The ServiceHost cannot register new models without restart — this is an intentional simplification

---

### 8. `events.proto` — enum conventions

**Decision:** Enum values are uppercase and prefixed with the type name: `MODEL_STATUS_LOADING`, `INFERENCE_STATUS_COMPLETED`, etc.

**Rationale:**
- Proto3 industry standard — prevents name collisions between different enum types
- `Grpc.Tools` strips the prefix in generated C# code — resulting in `ModelStatus.Loading`, `InferenceStatus.Completed`
- Proto3 requirement: every enum's first value must be `0` with an `_UNSPECIFIED` suffix — this is the default when the field is not set

---

### 9. `events.proto` — `google.protobuf.Timestamp` in every event

**Decision:** Every event contains an `occurred_at` or `started_at` / `completed_at` field of type `google.protobuf.Timestamp`.

**Rationale:**
- The time of an event is diagnostically valuable — useful for logging and performance measurement
- `google.protobuf.Timestamp` is a timezone-independent UTC timestamp — no string conversion needed
- Easily converted in C#: `timestamp.ToDateTime()`

---

### 10. `events.proto` — `ServiceReadyEvent` as the startup signal

**Decision:** `ServiceReadyEvent` contains `version` and `started_at`, and signals to the CLI that the ServiceHost is ready to receive commands.

**Rationale:**
- The CLI waits for this event during auto-start — it is the basis for the pipe-ready timeout
- The `version` field is useful for future compatibility checks — CLI and ServiceHost versions can be compared
- `started_at` is diagnostically valuable — shows how long startup took

---

### 11. `events.proto` — single `ModelStatusChangedEvent` for all model states

**Decision:** A single `ModelStatusChangedEvent` handles loading, loaded, unloaded, and failed states via the `ModelStatus` enum — rather than a separate message type per state.

**Rationale:**
- Fewer message types — simpler CLI-side processing
- State changes are naturally a sequence of events — `Loading` → `Loaded` or `Failed`
- The optional `message` field provides textual detail on failure

---

### 12. `events.proto` — `LoadedModelsListEvent` repeated field

**Decision:** `LoadedModelsListEvent` contains a `repeated string loaded_aliases` field for the list of loaded models.

**Rationale:**
- `repeated` is the protobuf list type — in C# it becomes `RepeatedField<string>`, usable as `IList<string>`
- Only loaded aliases are included — the human knows which models exist; only the loaded state matters at runtime
- No complex model object is needed — the alias is a sufficient identifier

---

### 13. `events.proto` — `InferenceProgressEvent` contents

**Decision:** `InferenceProgressEvent` contains `request_id`, `step_id`, `elapsed_seconds`, `status_message`.

**Rationale:**
- `elapsed_seconds`: simple progress indicator — the CLI displays "Generating... (Xs)"
- `status_message`: plain text status — e.g. "Generating... 30s | 412 tokens | 13.7 tok/s"
- Not token-by-token streaming — the Scaffold Protocol works from input files; there is no interactive typing experience requirement
- 30-second interval — sufficient feedback without flooding the event pipe

---

### 14. `events.proto` — `output_file_path` in `InferenceCompletedEvent`

**Decision:** `InferenceCompletedEvent` includes the `output_file_path` field.

**Rationale:**
- The CLI needs to know where the generated output file is located for human validation
- The ServiceHost determines the concrete filename (`{stepId}_{requestId[..8]}.md`) within `output_folder`
- The CLI passes this path to `IHumanValidationService` (and `IFileEditorLauncher`) to open the file
- This path is also written to the audit log

---

### 15. `events.proto` — `ServiceErrorEvent` for non-request-scoped errors

**Decision:** A dedicated `ServiceErrorEvent` type for errors not associated with a specific request, with `error_code` and `error_message` fields.

**Rationale:**
- Some errors cannot be tied to a `request_id` — e.g. `models.yaml` parse errors, pipe errors
- `error_code` for machine processing — the CLI can decide whether to stop or continue based on specific codes
- `error_message` is human-readable — can be printed to the console
- Inference-specific errors go into `InferenceFailedEvent` — `ServiceErrorEvent` is reserved for infrastructure-level errors

---

### 16. `commands.proto` — dedicated file for `CommandEnvelope`

**Decision:** `CommandEnvelope` lives in a dedicated `commands.proto` file that imports `inference.proto` and `models.proto`.

**Rationale:**
- `CommandEnvelope` has a single responsibility: aggregate all command types
- If it lived in `inference.proto`, that file would need to import `models.proto` — mixing concerns
- Clean dependency graph: `commands.proto` imports others, but nothing imports it — no circular dependencies

---

### 17. Field number consistency

**Decision:** Field numbers in all message types start at 1 and increment sequentially. No strategic field number allocation.

**Rationale:**
- The current protocol does not require backward compatibility with older versions — this is the v1.0 baseline
- If a field must be removed, mark it with `reserved` for both the number and the name:
  ```protobuf
  reserved 3;
  reserved "old_field_name";
  ```
- This prevents accidentally reusing a deleted field number

## Summary — proto file responsibilities

| File | Contents | Direction |
|---|---|---|
| `inference.proto` | InferRequest, CancelInferRequest, ShutdownRequest | CLI → ServiceHost |
| `models.proto` | LoadModelRequest, UnloadModelRequest, ListModelsRequest | CLI → ServiceHost |
| `commands.proto` | CommandEnvelope (imports inference + models) | CLI → ServiceHost |
| `events.proto` | EventEnvelope + 10 event types + 2 enums | ServiceHost → CLI |

## Dependency graph

```
commands.proto
    ├── imports inference.proto
    └── imports models.proto

events.proto
    └── imports google/protobuf/timestamp.proto

inference.proto   ← no imports
models.proto      ← no imports
```
