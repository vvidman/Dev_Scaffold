# ADR – Scaffold CLI Refactor

**Date:** 2026-03-05
**Updated:** 2026-03-22
**Status:** Accepted
**Affected projects:** Scaffold.CLI, Scaffold.Application

---

## Context

The original CLI implementation (based on `PipelineRunner`) managed the entire pipeline orchestration within a single continuous process — including model lifecycle. This approach was incompatible with the human-in-the-loop validation model, where:

- An arbitrary amount of time may pass between steps
- The output of one step is the input to the next — not known in advance
- Tasks are generated by step 1 — their number and content are determined at runtime
- Steps 2–4 run per task — not once, but once per generated task

The human is the orchestrator — they decide when the next step runs and with what input.

The second refactor round (2026-03-17) introduced the audit log system and structured console output, because:
- `Console.WriteLine` calls were scattered and unarchived — all trace was lost after a session
- The human validation decision (especially the rejection clarification) was not recorded anywhere
- Different levels of output (infrastructure, inference progress, validation) were not visually distinct

The third refactor round (2026-03-22) introduced Clean Architecture layering and SOLID principles:
- `ScaffoldSession` was split into `ScaffoldStepOrchestrator`, `InferenceResultHandler`, and `RefinementStrategy`
- All major components are now backed by interfaces — testable and replaceable
- `ConsoleHumanValidationService` became truly async and platform-independent via `IFileEditorLauncher`

---

## Decisions

---

### 1. CLI is step-scoped, not pipeline-scoped

**Decision:** The CLI does not run a pipeline — it runs a single step. Each CLI invocation executes one step and then exits. The ServiceHost keeps running.

**Rationale:**
- The inputs of pipeline steps depend on each other's outputs — they cannot be known in advance
- Human validation and orchestration are the human's responsibility — the CLI does not take over
- The concept of `pipeline.yaml` is eliminated — it would have been unnecessary complexity

**CLI commands:**
```bash
DevScaffold --step <step_name>
DevScaffold --step <step_name> --input <task_yaml_path>   # secondary task-specific input
DevScaffold --apply <folder> [<folder2>...] [--dry-run]   # copy artifacts/ → project_root
DevScaffold shutdown
```

---

### 2. CLI session lifecycle — one step, then exit

**Decision:** A CLI session lives as long as the CLI process runs. One invocation executes the step with human validation, then the CLI exits. The ServiceHost lifecycle is independent.

**Rationale:**
- The human decides when the next step runs — the CLI does not need to stay running
- Simpler and more predictable lifecycle
- The Reject → regenerate iteration stays within the session lifetime

**Lifecycle:**
```
DevScaffold --step x  →  ServiceHost starts (if not running) + step runs  →  CLI exits
DevScaffold --step x  →  ServiceHost already running + step runs           →  CLI exits
DevScaffold --step x  →  ServiceHost already running + step runs           →  CLI exits
DevScaffold shutdown  →  ServiceHost stops                                 →  CLI exits
```

---

### 3. ServiceHost lifecycle — independent of the CLI

**Decision:** The ServiceHost starts on the first `--step` invocation and runs until an explicit `shutdown` command is received. It does not stop when the CLI exits.

**Rationale:**
- Models stay in memory between steps — no need to reload on every invocation
- A human work session consists of multiple CLI invocations — the ServiceHost serves all of them
- An explicit shutdown command represents a clear and intentional stop

---

### 4. `pipeline.yaml` concept eliminated

**Decision:** There is no `pipeline.yaml`, `IPipelineConfigReader`, `PipelineConfig`, or `PipelineStepRef` in the CLI.

**Rationale:**
- The pipeline order lives in the human's mind and the protocol specification — not a config file
- A predefined step list contradicts the human-in-the-loop principle
- `PipelineRunner` and associated domain models have been removed from the CLI

---

### 5. CLI as a thin client

**Decision:** The CLI knows nothing about LLamaSharp, model management, or inference details. It only sends `CommandEnvelope` messages and receives `EventEnvelope` messages.

**Rationale:**
- Model lifecycle is the ServiceHost's responsibility
- The CLI is lightweight and starts quickly — no model loading wait
- The `Scaffold.Infrastructure.Inference` project reference was entirely removed from the CLI

**What remains in the CLI:**
- `IStepAgentConfigReader` — loads the agent system prompt
- `IInputAssembler` — input YAML assembly and path reference resolution
- `ConsoleHumanValidationService` — human validation interaction
- `FileAuditLogger` — audit log writing
- Generation number computation — determines the step output folder

---

### 6. Component structure

**Decision:** Responsibility is distributed across focused components following the Single Responsibility Principle:

| Component | Single responsibility |
|---|---|
| `PipeClient` | Named Pipe connection management (via `IPipeClient` interface) |
| `ServiceHostLauncher` | ServiceHost auto-start and ready-wait |
| `ScaffoldStepOrchestrator` | Retry loop control, request assembly, decision-based branching |
| `InferenceResultHandler` | Event subscription, automatic validation, human validation call |
| `RefinementStrategy` | Refinement prompt construction for auto and human reject |
| `FileAuditLogger` | Audit log writing to file, auto-flush |
| `Program.cs` | Command parsing, generation computation, DI composition root |
| `TaskBreakdownSplitter` | Post-processor: splits accepted markdown → per-task YAML files in `tasks/` |
| `CodingOutputExtractor` | Post-processor: extracts code blocks from accepted markdown → `artifacts/` |
| `DefaultMarkdownArtifactExtractor` | Regex-based fenced code block extraction with `filepath_hint_prefix` resolution |

---

### 7. `PipeClient` — `EventReceived` callback

**Decision:** `PipeClient` uses a `Func<EventEnvelope, Task>` delegate (`EventReceived`) to forward events, rather than `IObservable<EventEnvelope>` or a channel. `PipeClient` implements the `IPipeClient` interface — `ScaffoldStepOrchestrator` and `InferenceResultHandler` access it through the interface.

**Rationale:**
- Simpler than Rx or `System.Threading.Channels` — no extra dependencies
- The use case does not require backpressure or multiple concurrent subscribers
- `InferenceResultHandler` subscribes with one callback and unsubscribes after each attempt
- The `IPipeClient` interface enables testability and independence from CLI infrastructure

---

### 8. `WaitForReadyAsync` before the event loop

**Decision:** `PipeClient.WaitForReadyAsync` reads the event pipe directly, bypassing the event loop. The event loop only starts after `ServiceReadyEvent` has been received.

**Rationale:**
- `ServiceReadyEvent` is special — `ServiceHostLauncher` waits for it synchronously during startup
- If the event loop started first, a race condition could occur in processing the ready event
- Guaranteed order: WaitForReady → event loop starts → command pipe connects

---

### 9. `ServiceHostLauncher` — pipe existence check instead of process check

**Decision:** `ServiceHostLauncher` does not check whether the ServiceHost is running by process name — it tests for Named Pipe existence (`Connect(timeout: 0)`).

**Rationale:**
- Process name lookup may require elevated privileges and is platform-dependent
- If the pipe is alive, the ServiceHost is running and accepting connections — that is the relevant fact

**Two cases:**
- Pipe does not exist → start a new ServiceHost process
- Pipe exists but `ServiceReadyEvent` does not arrive → retry with reconnect attempt

---

### 10. Retry policy — 3 attempts, 60s timeout, no inter-attempt delay

**Decision:** Maximum 3 attempts, 60-second timeout per attempt waiting for `ServiceReadyEvent`. No extra delay between attempts.

**Rationale:**
- 60 seconds is sufficient for the ServiceHost to start (model loading is lazy — not at startup)
- The timeout itself is the wait — additional delay after it would be redundant
- 3 attempts is enough: repeated failure likely indicates a configuration problem

---

### 11. `TaskCompletionSource`-based inference result waiting

**Decision:** `InferenceResultHandler.HandleAsync` uses a single `TaskCompletionSource<ValidationDecision>`. The event handler switch branches (`InferenceCompleted`, `InferenceCancelled`, `InferenceFailed`) signal the result through the TCS. It subscribes to `IPipeClient.EventReceived` and always unsubscribes in a `finally` block — even on exception.

**Rationale:**
- An inference result is an asynchronous event — TCS is the most natural C# primitive for this
- A single TCS is sufficient: the three outcomes (completed/cancelled/failed) all map to `ValidationDecision`, with failed setting an exception on the TCS
- The `finally` block guarantees unsubscription — no memory leaks

---

### 12. Rejected output is not included in the regeneration prompt

**Decision:** The refinement prompt contains the original input and the clarification — not the text of the rejected output.

**Rationale:**
- If the AI starts from its own bad output, it tends to "patch" it rather than rethink
- The clarification alone is sufficient guidance for regeneration
- The rejected output remains on disk — the human can see it, but the AI does not receive it back

---

### 13. Shutdown — dedicated CLI command

**Decision:** The ServiceHost is shut down via a dedicated `DevScaffold shutdown` command, which sends a `ShutdownRequest` on the command pipe.

**Rationale:**
- The ServiceHost lifecycle is independent of the CLI lifecycle — implicit shutdown is inappropriate
- The human's deliberate decision to stop is better expressed by an explicit command
- Ctrl+C on the CLI stops only the CLI; the ServiceHost keeps running

---

### 14. `ConsoleHumanValidationService` — truly async, `IFileEditorLauncher`

**Decision:** `ValidateAsync` is a genuine async implementation: `Console.In.ReadLineAsync()` replaces the blocking `Console.ReadLine()`. File opening is delegated to `IFileEditorLauncher`. `DefaultFileEditorLauncher` uses `UseShellExecute = true` — the OS determines which editor is used based on the file extension (cross-platform).

**Rationale:**
- `Console.ReadLine()` blocks the threadpool thread — this is undesirable in an async context
- `IFileEditorLauncher` decouples platform-specific editor logic — testable and replaceable
- `UseShellExecute = true` works on Windows, macOS, and Linux — no hardcoded `notepad.exe`

---

### 15. Step output folder structure — generation-based numbering

**Decision:** Every `DevScaffold --step <n>` invocation gets a dedicated step output folder: `{output}/{pipe_name}/{stepId}_{n}`, where `n` is the current generation number (starting from 1).

```
output/
  MyProject/
    task_breakdown_1/     ← first run
      task_breakdown_abc12345.md
      audit.log
      tasks/              ← TaskBreakdownSplitter output
        task_01.yaml
        task_02.yaml
    task_breakdown_2/     ← after first rejection
      task_breakdown_def67890.md
      audit.log
    coding_1/
      coding_xyz99887.md
      audit.log
      artifacts/          ← CodingOutputExtractor output
        src/Services/FooService.cs
```

**Rationale:**
- Generations are traceable: how many regenerations were needed to reach an acceptable result
- The rejection clarification and the decision remain in the audit log — nothing is lost
- Each run is isolated: output and audit log are in one place
- The human can see the evolution — generations can be compared side by side

---

### 16. Generation computation — filesystem-based, process-restart-safe

**Decision:** The CLI computes the generation number at runtime by counting existing `{stepId}_*` directories in the output folder and taking the maximum number + 1.

**Rationale:**
- The CLI process is short-lived — it cannot hold state in memory between runs
- Filesystem-based computation remains correct after process restarts
- Sparse sequences (e.g. if the human deletes a folder) are handled correctly: the max is taken, not the count

---

### 17. Audit log — plain text, fixed-width tag column, auto-flush

**Decision:** Every step output folder contains an `audit.log` file. Format:
```
2026-03-22 14:23:01.123 [INFO ] [SESSION_START   ] step=task_breakdown generation=3
2026-03-22 14:23:01.124 [INFO ] [CONFIG          ] model=qwen-7b system_prompt_length=342 max_tokens=4096
2026-03-22 14:25:43.891 [INFO ] [INFERENCE_DONE  ] tokens=847 elapsed=162s tok_s=5.2
2026-03-22 14:25:50.012 [INFO ] [VALIDATION      ] outcome=Reject clarification="More detail needed"
2026-03-22 14:25:50.013 [INFO ] [SESSION_END     ] total_elapsed=169s outcome=Reject attempts=2
```

The log is written with `AutoFlush = true`.

**Rationale:**
- Human-readable: no external tool needed
- Parser-friendly: fixed-position timestamp (23 chars), level (7 chars), tag (18 chars) — positional split is sufficient
- `key=value` format is also suitable for statistical processing (e.g. average tok/s per step type)
- Auto-flush: partial data is preserved on CLI crash — the most important information (task issued, decision made) is not lost

**Recorded events:**

| Tag | Contents |
|---|---|
| `SESSION_START` | step, generation |
| `CONFIG` | model, system_prompt_length, max_tokens, output_folder |
| `INFERENCE_START` | request_id, step, model, attempt |
| `INFERENCE_DONE` | tokens, elapsed, tok_s |
| `OUTPUT` | path (full path of the generated file) |
| `VALIDATION` | outcome, clarification (on Reject) |
| `SESSION_END` | total_elapsed, outcome, attempts |
| `ERROR` | reason, message |

---

### 18. `IScaffoldConsole` — three-level console output with colour coding

**Decision:** All components use `IScaffoldConsole` instead of `Console.WriteLine`, providing three-level, colour-coded output.

| Level | Method | Colour | Content |
|---|---|---|---|
| CLI | `WriteCli` | Cyan | Infrastructure, ServiceHost startup, pipe, folder creation |
| Session | `WriteSession` | Gray | Inference progress, ServiceHost events |
| Validation | `WriteValidation` | Yellow | Human validation interaction, decision prompts |
| Error | `WriteError` | Red | Errors (written to stderr) |

**Rationale:**
- Different output levels are visually distinct — the human can see at a glance what is infrastructure and what is AI output status
- The abstraction makes console output testable
- The three levels naturally map to the three layers of the system: CLI infrastructure, inference execution, human decision

## Summary — original vs. refactored CLI

| Aspect | Original CLI | 1st refactor | 2nd refactor | 3rd refactor | 4th refactor |
|---|---|---|---|---|---|
| Granularity | Pipeline-scoped | Step-scoped | Step-scoped | Step-scoped | Step-scoped |
| Orchestration | PipelineRunner | Human | Human | Human | Human |
| Model management | In CLI | ServiceHost | ServiceHost | ServiceHost | ServiceHost |
| Step order | pipeline.yaml | Human decides | Human decides | Human decides | Human decides |
| CLI lifecycle | Runs until pipeline end | Exits after one step | Exits after one step | Exits after one step | Exits after one step |
| ServiceHost lifecycle | Tied to CLI | Independent | Independent | Independent | Independent |
| Communication | Direct dependency | Named Pipe | Named Pipe | Named Pipe | Named Pipe |
| Output folder | ServiceHost decides | ServiceHost decides | CLI decides (generation) | CLI decides (generation) | CLI decides (generation) |
| Audit log | None | None | File, auto-flush | File, auto-flush | File, auto-flush |
| Console output | Console.WriteLine | Console.WriteLine | Colour-coded | Colour-coded | Colour-coded |
| Session orchestrator | PipelineRunner | ScaffoldSession | ScaffoldSession | ScaffoldStepOrchestrator + InferenceResultHandler + RefinementStrategy | (unchanged) |
| Editor integration | None | None | None | IFileEditorLauncher (cross-platform) | (unchanged) |
| Post-processing | None | None | None | None | IStepPostProcessor pipeline + PostProcessorContext |
| Artifact extraction | None | None | None | None | IMarkdownArtifactExtractor → artifacts/ |
| Secondary input | None | None | None | None | --input flag (appended to project_context) |
| Apply command | None | None | None | None | --apply copies artifacts/ → project_root |
